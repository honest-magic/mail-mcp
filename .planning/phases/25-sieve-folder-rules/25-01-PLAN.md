# Phase 25 Plan 01: SIEVE Folder Rules

## Objective

Add four MCP tools (`list_filters`, `get_filter`, `set_filter`, `delete_filter`) that let AI agents manage server-side SIEVE email filters via the ManageSieve protocol (RFC 5804). Implement a `SieveClient` class using raw TLS sockets — no external library required.

## Context

@.planning/phases/25-sieve-folder-rules/CONTEXT.md

## Tasks

### Task 1 — Add manageSievePort to account schema (type="auto")

Add `manageSievePort: z.number().int().positive().optional()` to the `emailAccountSchema` in `src/config.ts`. This allows users to override the default ManageSieve port (4190).

**Done criteria:**
- `emailAccountSchema` accepts `manageSievePort` as optional number
- Existing accounts without `manageSievePort` still parse successfully

### Task 2 — Implement SieveClient with failing tests (type="auto", tdd="true")

Create `src/protocol/sieve.test.ts` with failing tests for:
- `listScripts()` — parses multi-script response with active marker
- `getScript(name)` — parses literal string response
- `putScript(name, content)` — sends literal string upload
- `deleteScript(name)` — sends DELETE command
- Authentication: PLAIN SASL base64 encode
- Error: graceful failure message when connection refused

All tests must fail (RED) before implementation begins.

**Behavior spec:**
```typescript
interface SieveScript {
  name: string;
  active: boolean;
}

class SieveClient {
  constructor(host: string, port: number, user: string, password: string)
  connect(): Promise<void>      // TLS connect + AUTHENTICATE
  disconnect(): Promise<void>   // LOGOUT
  listScripts(): Promise<SieveScript[]>
  getScript(name: string): Promise<string>
  putScript(name: string, content: string): Promise<void>
  deleteScript(name: string): Promise<void>
}
```

### Task 3 — Implement SieveClient (GREEN phase) (type="auto", tdd="true")

Implement `src/protocol/sieve.ts` to make all tests pass.

Key implementation details:
- Use `tls.connect()` from Node.js `node:tls`
- Parse server greeting to extract SASL mechanisms
- Authenticate using PLAIN SASL: `\0username\0password` → base64
- Handle literal strings: `{N+}\r\n` followed by N bytes for uploads; `{N}\r\n` for downloads
- Parse `LISTSCRIPTS` response: lines ending in `ACTIVE` marker
- All responses terminated by `OK` or `NO` (with optional quoted message)
- Timeout: 10 seconds connection timeout
- `rejectUnauthorized: false` to support self-signed certs on self-hosted servers

### Task 4 — Add 4 SIEVE tools to index.ts (type="auto")

**Tool definitions** (add to `getTools()` array in `src/index.ts`):

```
list_filters   — readOnlyHint: true
get_filter     — readOnlyHint: true
set_filter     — readOnlyHint: false, destructiveHint: true
delete_filter  — readOnlyHint: false, destructiveHint: true
```

Add `set_filter` and `delete_filter` to the `WRITE_TOOLS` Set.

**Dispatch handlers** — add to `dispatchTool()` in `src/index.ts`:
- `list_filters`: create SieveClient → connect → listScripts → disconnect → return JSON
- `get_filter`: create SieveClient → connect → getScript → disconnect → return content
- `set_filter`: create SieveClient → connect → putScript → disconnect → return OK message
- `delete_filter`: create SieveClient → connect → deleteScript → disconnect → return OK message

Pattern: SieveClient is created per-request (stateless, short-lived) using account credentials loaded via `loadCredentials(account.id)`.

Also add `set_filter` and `delete_filter` to the setupToolHandlers section following the same pattern as other write tools.
