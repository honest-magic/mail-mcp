# Phase 30 Plan 01: Per-Tool Allow-Listing

## Objective

Add a `--allow-tools` CLI flag that lets users specify exactly which write tools are available, instead of the binary all-or-nothing `--read-only` mode. Read tools are always available. `--read-only` and `--allow-tools` are mutually exclusive.

## Context

@.planning/phases/30-per-tool-allow-listing/CONTEXT.md

## Tasks

### Task 1 — Write failing tests (TDD RED)
type="auto" tdd="true"

Create `src/index.allowlist.test.ts` with failing tests covering:
- AL-01: `getTools()` with `allowedTools` set to `['send_email']` returns read tools + only `send_email` (not other write tools)
- AL-02: `getTools()` with `allowedTools` set to `['send_email', 'create_draft']` returns exactly read tools + those 2
- AL-03: `getTools()` with empty `allowedTools` set returns only read tools (no write tools)
- AL-04: `dispatchTool()` with `allowedTools=['send_email']` allows `send_email` (no isError)
- AL-05: `dispatchTool()` with `allowedTools=['send_email']` blocks `delete_email` (isError: true)
- AL-06: `dispatchTool()` with `allowedTools=['send_email']` blocks `move_email` (isError: true)
- AL-07: Constructor with `readOnly=true` and `allowedTools` set throws an error
- AL-08: `getTools()` with no allowedTools (undefined) and readOnly=false returns all 29 tools (backward compat)
- AL-09: Server instructions mention "allow-listed" tools when `allowedTools` is set
- AL-10: `dispatchTool()` with `allowedTools` — allowed tool returns allowed-specific error message, not read-only message

### Task 2 — Implement allow-listing in MailMCPServer (TDD GREEN)
type="auto" tdd="true"

Modify `src/index.ts`:
1. Constructor: add `allowedTools?: Set<string>` parameter (after `readOnly`); throw `Error` if both `readOnly=true` and `allowedTools` is provided
2. `getTools()`: signature becomes `getTools(readOnly: boolean, allowedTools?: Set<string>)`. Logic:
   - If `readOnly` → return only read tools (existing behavior)
   - If `allowedTools` is set → return read tools + write tools in `allowedTools`
   - If neither → return all tools
3. `dispatchTool()`: add `allowedTools?: Set<string>` as 3rd param (or use instance field). Guard: if tool is write AND `allowedTools` is set AND tool not in `allowedTools` → return `isError: true` with message: `"Tool '{name}' is not available: not in the allowed tools list. Allowed write tools: {list}."`
4. `setupToolHandlers()`: update `getTools()` call to pass `this.allowedTools`
5. `setupToolHandlers()` CallToolRequestSchema handler: add allow-list guard (parallel to read-only guard)
6. Instructions: if `allowedTools` is set, add sentence listing the enabled write tools
7. `main()`: parse `--allow-tools` as string option, split by comma, validate mutual exclusivity with `--read-only` (print error + exit 1), construct `MailMCPServer` with the parsed `Set`
8. `--help`: add `--allow-tools` documentation
