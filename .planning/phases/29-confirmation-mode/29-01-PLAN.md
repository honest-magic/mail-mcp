# Objective

Implement optional confirmation mode (`--confirm` CLI flag). When active, write tool calls return a `confirmationRequired` response with a UUID instead of executing. The second call — same tool + `confirmationId` — executes the action. Pending confirmations expire after 5 minutes.

# Tasks

## Task 1 — Create ConfirmationStore utility (TDD)
type="auto" tdd="true"

### Behavior

`ConfirmationStore` class in `src/utils/confirmation-store.ts`:
- `CONFIRMATION_TTL_MS = 5 * 60 * 1000` (exported constant)
- Constructor: `new ConfirmationStore(ttlMs?: number)` — injectable TTL for tests
- `create(toolName: string, args: Record<string, unknown>): string` — stores pending confirmation, returns UUID
- `consume(id: string): PendingConfirmation | undefined` — returns the confirmation if valid and not expired, then **removes** it (single-use)
- `size` getter — raw store count (includes stale entries; same semantics as MessageBodyCache)
- `PendingConfirmation` interface (exported): `{ toolName: string; args: Record<string, unknown>; createdAt: number; ttlMs: number }`
- Expired entries removed on `consume()` (lazy eviction, consistent with MessageBodyCache pattern)
- `consume()` on expired ID returns `undefined` and removes the entry

### Implementation notes

- Use `crypto.randomUUID()` (built-in `node:crypto` — no imports needed from v14.17+)
- TTL check: `Date.now() - entry.createdAt >= entry.ttlMs` means expired

Test file: `src/utils/confirmation-store.test.ts`

## Task 2 — Wire --confirm flag into MailMCPServer (TDD)
type="auto" tdd="true"

### Behavior

- `MailMCPServer` constructor accepts second param: `confirmMode: boolean = false`
- Server instructions append confirmation mode message when active (mirrors read-only instruction pattern)
- `MailMCPServer` holds a `private readonly confirmStore = new ConfirmationStore()` instance
- All write tool schemas gain an optional `confirmationId` field: `{ type: 'string', description: 'Confirmation token from a prior confirmation-required response. Required to execute write operations when server is in --confirm mode.' }`
- In `setupToolHandlers` (CallToolRequestSchema handler): when `confirmMode` is true and tool is in WRITE_TOOLS:
  - If `args.confirmationId` is present: call `confirmStore.consume(id)`. If valid, strip `confirmationId` from args and execute as normal. If invalid/expired: return error `{ isError: true, text: "Confirmation token invalid or expired. Call the tool again without confirmationId to get a new token." }`
  - If `args.confirmationId` is absent: call `confirmStore.create(toolName, args)` and return `{ confirmationRequired: true, action: toolName, description: <human-readable summary>, confirmationId: <uuid>, expiresIn: "5 minutes" }` — do NOT execute the action
- Read tools execute immediately regardless of confirmMode

Human-readable description format per tool:
- `send_email`: `"Send email to {to} with subject '{subject}'"`
- `create_draft`: `"Save draft to {to} with subject '{subject}'"`
- `reply_email`: `"Reply to email UID {uid} in {folder}"`
- `forward_email`: `"Forward email UID {uid} to {to}"`
- `delete_email`: `"Permanently delete email UID {uid} from {folder}"`
- `move_email`: `"Move email UID {uid} from {sourceFolder} to {targetFolder}"`
- `batch_operations`: `"Batch {action} {uids.length} emails in {folder}"`
- `modify_labels`: `"Modify labels on email UID {uid} in {folder}"`
- `mark_read`: `"Mark email UID {uid} as read in {folder || 'INBOX'}"`
- `mark_unread`: `"Mark email UID {uid} as unread in {folder || 'INBOX'}"`
- `star`: `"Star email UID {uid} in {folder || 'INBOX'}"`
- `unstar`: `"Unstar email UID {uid} in {folder || 'INBOX'}"`
- `register_oauth2_account`: `"Register OAuth2 credentials for account {accountId}"`
- `set_filter`: `"Create/update SIEVE filter '{name}' on account {accountId}"`
- `delete_filter`: `"Delete SIEVE filter '{name}' on account {accountId}"`

Tests should cover: first call returns confirmationRequired, second call with valid ID executes, expired ID returns error, wrong tool on second call handled by ID lookup (IDs are opaque — tool name mismatch is fine as long as the stored args get replayed correctly).

Test file: `src/index.confirm.test.ts`
