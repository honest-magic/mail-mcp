## Critical Gaps (Blockers)

### Gap 1: SMTP-04 — Sent-folder APPEND Broken at Runtime

**File:** `src/services/mail.ts` line 45
**Issue:** `info.message.toString()` — nodemailer's `SentMessageInfo` return type does not have a `.message` property. This throws a `TypeError` which is silently caught at lines 48–50. The `appendMessage('Sent', ...)` call is never reached.
**Symptom:** `send_email` tool returns `"Email successfully sent ... and saved to Sent folder."` even though the Sent-folder copy is never created.
**Fix:** Construct the raw MIME message before sending (e.g., using nodemailer's `getMessage()` or composing separately), then pass to `appendMessage`.

### Gap 2: THRD-01/02/03 — Thread Retrieval Gmail-Only

**File:** `src/protocol/imap.ts` lines 147–151
**Issue:** `fetchThreadMessages` only searches by `{ 'x-gm-thrid': threadId }`, a Gmail-proprietary IMAP extension. For all other providers (Outlook, Fastmail, standard IMAP), the search throws and is caught, returning `[]` silently.
**Symptom:** `get_thread` returns an empty thread for all non-Gmail accounts.
**Fix:** Implement RFC 5256 `THREAD` command as primary strategy, fall back to `References`/`In-Reply-To` header matching, then try `x-gm-thrid` as a provider-specific optimization.

### Gap 3: IMAP-01 — Snippet Field Always Empty

**File:** `src/protocol/imap.ts` lines 79, 111, 163
**Issue:** `snippet` is set to `''` with a comment noting it as a placeholder. The requirement specifies "list recent messages with **snippets**".
**Symptom:** `list_emails` returns messages with empty snippet fields — callers cannot get a preview without calling `read_email`.
**Fix:** Fetch a partial body preview (e.g., first 200 chars of text/plain part) during `listMessages` using imapflow's bodyPart fetch.
