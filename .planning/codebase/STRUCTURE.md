## Directory Purposes

**src/cli/**
- Purpose: Command-line interface implementation for account setup and management
- Contains: Account CRUD operations (add, list, remove), interactive prompts
- Key files: `accounts.ts`

**src/protocol/**
- Purpose: Email protocol implementations with low-level server communication
- Contains: IMAP client (message fetch, search, move, label, thread), SMTP client (send)
- Key files: `imap.ts`, `smtp.ts`

**src/security/**
- Purpose: Authentication and credential security
- Contains: OS keychain integration, OAuth2 token refresh logic
- Key files: `keychain.ts`, `oauth2.ts`

**src/services/**
- Purpose: High-level mail operations and business logic
- Contains: MailService class orchestrating IMAP/SMTP, email read/send/search, attachments, batch operations
- Key files: `mail.ts`

**src/types/**
- Purpose: Shared TypeScript type definitions
- Contains: EmailAccount, Credentials, AuthType, interfaces
- Key files: `index.ts`

**src/utils/**
- Purpose: Utility functions and transformations
- Contains: HTML to Markdown conversion
- Key files: `markdown.ts`

**dist/**
- Purpose: Compiled TypeScript output
- Contains: Generated .js and .d.ts files (generated by tsc)
- Committed: No (generated on build)

## Key File Locations

- `src/index.ts`: Main MCP server entry point, MailMCPServer class, tool routing, CLI detection

**Testing:**

## Naming Conventions

**Files:**
- Protocol clients: `[protocol].ts` (e.g., `imap.ts`, `smtp.ts`)
- Classes: PascalCase (e.g., `MailService`, `ImapClient`, `SmtpClient`)
- Test files: `[module].test.ts` (e.g., `mail.test.ts`, `imap.test.ts`)
- Utilities: kebab-case or descriptive (e.g., `markdown.ts`)

**Directories:**
- Feature-based organization: `cli/`, `protocol/`, `security/`, `services/`, `types/`, `utils/`
- No deep nesting: Max 2 levels (e.g., `src/protocol/imap.ts` not `src/protocol/imap/client.ts`)

**Functions & Variables:**
- Operations: camelCase (e.g., `listEmails()`, `searchMessages()`, `getValidAccessToken()`)
- Constants: UPPER_SNAKE_CASE (e.g., `ACCOUNTS_PATH`, `WRITE_TOOLS`)
- Interfaces: PascalCase (e.g., `EmailAccount`, `MessageMetadata`, `OAuth2Tokens`)

**MCP Tool Names:**
- kebab-case (e.g., `list_accounts`, `read_email`, `send_email`, `get_thread`, `extract_attachment_text`)

## Where to Add New Code

**New Feature (Email Operation):**
- Primary code: Add method to `MailService` in `src/services/mail.ts`
- Protocol layer: Add underlying method to `ImapClient` or `SmtpClient` in `src/protocol/[protocol].ts` if needed
- MCP exposure: Add tool definition to `getTools()` and handler in `setupToolHandlers()` in `src/index.ts`
- Tests: Add tests to `src/services/mail.test.ts` and protocol tests

**New Account Management Command:**
- Implementation: Add handler function in `src/cli/accounts.ts` (e.g., `updateAccount()`)
- Routing: Add case to switch statement in `handleAccountsCommand()`
- Tests: Add tests to `src/cli/accounts.test.ts`

**New Protocol Feature (IMAP/SMTP):**
- Implementation: Add method to `ImapClient` or `SmtpClient` in `src/protocol/[protocol].ts`
- Lock management: Acquire mailbox lock before IMAP operations, release in finally block
- Error handling: Throw Error with descriptive message (will be caught in MCP handler)
- Tests: Add tests to `src/protocol/[protocol].test.ts`

**New Utility:**
- Implementation: Add function to existing utility file in `src/utils/` or create new `src/utils/[name].ts`
- Export: Export from `src/utils/` and import where needed
- Tests: Add tests to `src/utils/[name].test.ts` if complex

**New Type Definition:**
- Implementation: Add to `src/types/index.ts` for shared types, or define locally in feature file for private types
- Naming: Use PascalCase for interfaces/types
- Exports: Export from `src/types/index.ts` if shared

## Cross-Module Dependencies

**Import patterns:**
- Protocol → types: `import { EmailAccount, Credentials } from '../types/index.js'`
- Service → protocol: `import { ImapClient, MessageMetadata } from '../protocol/imap.js'`
- Service → security: `import { loadCredentials, saveCredentials } from '../security/keychain.js'`
- Index → service: `import { MailService } from './services/mail.js'`
- Index → config: `import { getAccounts, saveAccounts } from './config.js'`
- CLI → config: `import { getAccounts, saveAccounts } from '../config.js'`
- Protocol → security: `import { loadCredentials } from '../security/keychain.js'` (IMAP, SMTP)
- OAuth2 → keychain: `import { loadCredentials, saveCredentials } from './keychain.js'`

**Circular dependency avoidance:**
- Config module is pure (no imports from services/protocol)
- Types module has no dependencies (only exports)
- Security modules import from config only
- Services import from protocol and security, not reverse
