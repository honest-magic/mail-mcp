## Task 1: Fix SMTP-04 — Sent-folder APPEND

**File:** `src/services/mail.ts`

**Problem:** `sendEmail` calls `info.message.toString()` but nodemailer's `SentMessageInfo`
has no `.message` property. The TypeError is silently caught, so `appendMessage` never runs.

**Fix:** Build the raw RFC-822 message manually before calling `smtpClient.send()`,
using the same `\r\n`-joined header approach already used in `createDraft`. Pass that
pre-built string to `appendMessage('Sent', ...)`.

```typescript
// Replace lines 43-51 in mail.ts with:
async sendEmail(to: string, subject: string, body: string, isHtml: boolean = false, cc?: string, bcc?: string): Promise<any> {
  // Build raw message before sending so we can append to Sent folder
  const rawMessage = [
    `From: ${this.account.user}`,
    `To: ${to}`,
    ...(cc ? [`Cc: ${cc}`] : []),
    ...(bcc ? [`Bcc: ${bcc}`] : []),
    `Subject: ${subject}`,
    'MIME-Version: 1.0',
    `Content-Type: ${isHtml ? 'text/html' : 'text/plain'}; charset=utf-8`,
    '',
    body
  ].join('\r\n');

  const info = await this.smtpClient.send(to, subject, body, isHtml, cc, bcc);
  try {
    await this.imapClient.appendMessage('Sent', rawMessage, ['\\Seen']);
  } catch (e) {
    console.error('Failed to append to Sent folder:', e);
  }
  return info;
}
```

**Verify:** `rawMessage` is a string; `appendMessage` call is reached (no `.message` access).

## Task 2: Fix THRD-01/02/03 — Thread fallback for non-Gmail

**File:** `src/protocol/imap.ts`

**Problem:** `fetchThreadMessages` only uses `x-gm-thrid` (Gmail proprietary). For all
other providers, the search throws and `[]` is returned silently.

**Also fix in `src/services/mail.ts`:** `readEmail` exposes `x-gm-thrid` but not
`Message-ID`, so callers on non-Gmail have no threadId to pass to `get_thread`.

**Fix A — `imap.ts` `fetchThreadMessages`:**

After the x-gm-thrid try/catch, if `uids` is empty, add a second attempt using
`References` header search and `Message-ID` exact match:

```typescript
// After the x-gm-thrid block, if uids is still empty:
if (!uids || uids.length === 0) {
  try {
    const refUids = await this.client.search(
      { header: ['References', threadId] }
    ) as number[];
    const rootUids = await this.client.search(
      { header: ['Message-ID', threadId] }
    ) as number[];
    uids = [...new Set([...(refUids || []), ...(rootUids || [])])];
  } catch (e2) {
    // header search not supported — return empty
    return [];
  }
}
```

**Fix B — `mail.ts` `readEmail`:**

After the existing `x-gm-thrid` check, also expose `message-id`:

```typescript
const messageId = parsed.messageId || parsed.headers.get('message-id');
if (messageId) {
  header += `**Message-ID:** ${messageId}\n`;
}
```

**Verify:** Non-Gmail path: x-gm-thrid fails → falls back to header search → returns messages.
Gmail path: x-gm-thrid succeeds → header fallback skipped.

## Task 3: Fix IMAP-01 — Snippet field always empty

**File:** `src/protocol/imap.ts`

**Problem:** `listMessages` and `searchMessages` set `snippet: ''` permanently.
Requirement IMAP-01 specifies "list recent messages with snippets".

**Fix:** Add `bodyParts: ['TEXT']` to the imapflow `fetch` call in both methods.
Extract `msg.bodyParts?.get('TEXT')`, decode as UTF-8, collapse whitespace, truncate to 200 chars.

```typescript
// In listMessages fetch:
for await (const msg of this.client.fetch(range, {
  envelope: true, flags: true, internalDate: true, bodyParts: ['TEXT']
})) {
  const textBuf = (msg as any).bodyParts?.get('TEXT');
  const snippet = textBuf
    ? textBuf.toString('utf-8').replace(/\s+/g, ' ').slice(0, 200).trim()
    : '';
  messages.push({ ..., snippet });
}

// Same pattern for searchMessages fetch.
```

Note: `fetchThreadMessages` also sets `snippet: ''` but already fetches full messages via
`fetchMessageBody` when the caller needs content. Leave it as empty string there — the
thread view is metadata-only by design (THRD-03: token-efficient).

**Verify:** `snippet` field is non-empty for messages with text bodies.
